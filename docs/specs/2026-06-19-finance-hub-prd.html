1Executive Summary / Ringkasan Eksekutif

Finance Hub adalah sistem manajemen keuangan internal berbasis web untuk Account Payable, Petty Cash, Beasiswa, dan proses pembayaran yang dijalankan secara lokal (localhost/LAN). Dirancang sebagai platform multi-perusahaan untuk dua entitas: Sinar Mas Tjipta (SMT) dan Eka Tjipta Foundation (ETF).

Transformasi: Dari Excel ke Python

Finance Hub menggantikan dua sistem Excel yang sebelumnya berjalan terpisah dan tidak terhubung:

Sistem LamaDigantikan OlehKeterangan
DbBeasiswa v2.2.xlsm (Excel VBA Macro) Modul Beasiswa + PAM Database siswa, budget, payment, PAM — sebelumnya di shared folder Y:/
PA Excel manual (Microsoft Excel) Modul ETF Payment Application Tracking Payment Application per siswa — dikelola manual per tipe (AGRI/APP/LAND/SETF)
Integrasi Kunci: Selain mendigitalisasi kedua sistem secara terpisah, Finance Hub menghubungkan alur PA → Beasiswa sehingga Payment Application menjadi sumber tunggal yang sinkron ke Beasiswa dan PAM secara atomik. Lihat Bagian 2.6.

Posisi dalam Masterplan Automasi PT Sinar Mas Tjipta

Finance Hub adalah implementasi Fase 4 (F4) — AP, Tax & Payment. Bekerja dalam ekosistem bersama PR Portal (F2) sebagai hulu dan Accounting Hub (F5) sebagai hilir.

2Problem Statement / Latar Belakang & Tujuan

2.1 Masalah Sebelumnya

Sistem A — Beasiswa (DbBeasiswa v2.2.xlsm)

Sebelum — Excel VBA Macro di shared folder Y:/
  • File lock conflict — hanya 1 orang bisa edit bersamaan
  • Tidak ada autentikasi — siapapun yang akses Y:/ bisa ubah data
  • Tidak ada audit trail — tidak diketahui siapa/kapan mengubah data
  • PAM dibuat manual: isi sheet Detail PAM & Rangkuman PAM, cetak manual
  • PAM Recon via PivotTable — harus di-refresh manual
  • SLA dates (Days of PAM) dicatat di kolom Cat3–Cat7, tidak ada validasi format
  • Data Medical (DbHospital) tidak terintegrasi dengan rekap kategori
  • Data SMT & ETF tidak terpisah secara sistematis

Sistem B — Payment Application (Excel Manual)

Sebelum — PA Excel manual per tipe
  • Tidak ada tracking terstruktur per tipe PA (AGRI/APP/LAND/SETF)
  • 7 SLA dates per PA dicatat ad-hoc, tidak ada validasi atau monitoring
  • Nomor PAM diisi manual, rawan typo/duplikat
  • Tidak ada status tracking — sulit tahu PA mana yang open/on_process/complete
  • Data tidak terhubung ke Beasiswa — pembayaran harus diinput ulang di DbBeasiswa (duplikasi entri, rawan desync)
  • Tidak ada summary/agregasi — rekap per tipe/pillar harus dibuat manual

Masalah Lintas Sistem

Input ganda: Pembayaran harus diinput dua kali — sekali di PA Excel, sekali lagi di DbBeasiswa. Sumber kebenaran tidak jelas.

PAM tidak otomatis: Staf harus membuat PAM di DbBeasiswa secara manual setelah PA dibuat.

Rekonsiliasi manual: Antara PA Excel dan DbBeasiswa harus diselaraskan secara berkala.

2.2 Solusi

Portal terpusat

Semua transaksi finansial diinput via web app. Tidak ada Excel manual.

Multi-company

Data SMT dan ETF terisolasi penuh via company_id di semua tabel.

Auth berbasis JWT

Login wajib, role-based access, audit trail created_at/updated_at.

PA → Beasiswa atomik

Input PA → secara otomatis membuat record Beasiswa + PAM dalam satu transaksi DB.

PAM otomatis

Draft Memo auto-generate dari data payment; export PDF/Excel standar perusahaan.

Pillar-based PA

Tab per pillar (AGRI/APP/LAND/SETF); Summary view; filter jenjang, tahun, bulan.

Days of PAM

4 SLA dates terstruktur per PAM; bulk-update; monitoring keterlambatan.

REST API

Semua operasi tersedia via JSON API untuk integrasi sistem induk.

2.6 Integrasi PA → Beasiswa — Data Sinkron dari Sumber

Salah satu terobosan utama Finance Hub v1.x adalah menghubungkan Payment Application dengan Beasiswa sehingga satu tindakan di PA secara otomatis menciptakan record di Beasiswa dan PAM tanpa input ulang.

Sebelum — Dua Sistem Terpisah
  • Buat PA di Excel → catat 7 SLA dates manual
  • Ubah status PA manual (warna cell)
  • Tulis nomor PAM manual di DbBeasiswa
  • Input payment ulang per siswa di DbBeasiswa
  • Buat PAM ulang di sheet Detail PAM
  • ↑↑ Dua sumber, rawan desync ↑↑
Sesudah — Finance Hub Terintegrasi
  • Buat PA di ETF PA module → 7 SLA dates terstruktur
  • Input PA panel: pilih siswa dari PA open otomatis tersaring
  • "Simpan PAM" → satu klik, atomik:
  • ✅ PAM terbentuk otomatis
  • ✅ Beasiswa terbentuk dari PA (tidak input ulang)
  • ✅ Status PA → on_process + nomor PAM terisi
  • ↑↑ Satu sumber kebenaran ↑↑

Alur Detail: PA sebagai Sumber Kebenaran

LangkahSistem LamaFinance Hub
Buat Payment ApplicationIsi form Excel, simpan ke sheetBuat PA via web; tersimpan ke DB dengan 7 SLA dates
Proses pembayaranInput ulang ke DbBeasiswaPilih dari PA open di panel Input PA (auto-filter tipe)
Buat PAM numberTulis manual di DbBeasiswaAuto-generate dari sequence + tipe
Sinkronisasi dataManual, rawan desyncAtomik dalam satu transaksi DB
Status PATidak ada / manualopen → on_process → complete auto-update
Rekap per pillarManual PivotTableTab AGRI/APP/LAND/SETF + Summary tab otomatis

save_pa_payment() — Atomic Transaction

BEGIN TRANSACTION
  INSERT INTO pam_records (pam_no, source, total, ...) → PAM terbentuk
  INSERT INTO payment_beasiswa (per siswa, status='open') → Beasiswa terbentuk
  UPDATE {type}_pa SET
    nomor_pam = pam_no,
    status = 'on_process' → PA terupdate
COMMIT -- atau ROLLBACK jika error → tidak ada record parsial

Pillar ETF (Sudah Berjalan) → SMT (Roadmap)

Saat ini integrasi PA→Beasiswa berjalan untuk Eka Tjipta Foundation dengan 4 pillar:

AGRI APP LAND SETF → model siap direplikasi ke SMT (roadmap)

Begitu ETF stabil, model yang sama akan direplikasi untuk Sinar Mas Tjipta dengan pillar dan tipe PA yang sesuai struktur SMT. Data tetap terisolasi per company_id.

4.3 Alur ETF Payment Application (AGRI/APP/LAND/SETF)

Dashboard ETF → Menu: ETF Payment Application
  → Pilih tab: [AGRI] | [APP] | [LAND] | [SETF]
  → Daftar PA: satu baris = satu PA header

      → Badge: Open (abu) / On Process (kuning) / Complete (hijau)
  → Buat PA Baru → isi header + "+ Tambah Baris" per siswa → confirmModal
  → Edit PA (tombol Edit) → modal SLA dates + status update
      → status → on_process + nomor_pam kosong → auto-generate PAM number
  → Tab: Summary → pa_summary view; filter Tahun PA / Bulan PA / Nama / Status
  → Bulk Update SLA → checkbox baris → isi tanggal → confirmModal

4.4 Alur PAM Input Integration (PA → Payment Memo)

Core integration flow: PA menjadi sumber data; tidak perlu input ulang di Beasiswa atau PAM. 
Dashboard ETF → Menu: Payment Memo → Tab: Input PA

  → Pilih Tipe PAM: [AGRI] | [APP] | [LAND] | [SETF]
  → Pilih Transaksi Type: [Beasiswa] | [Klaim Medis] | [Others]
  → No. PAM: auto-generate readonly (PAM-054-AGRI-06-2026)
     ★ Override manual → format regex divalidasi real-time
     ★ Saat blur → collision check ke /payment-memo/pam/check

  → "+ Tambah Baris":
     → Cari siswa (hanya siswa dengan open PA tipe terpilih)
     → 1 PA match → auto-fill Amount, SLA dates
     → 2+ PA match → PA Picker sub-row (mini-table: No PA | Amount)

  → "Simpan PAM {TYPE}" → confirmModal
     Backend (atomik):
       1. INSERT pam_records (source='etf_{tab}')
       2. INSERT payment_beasiswa (per baris, status='open')
       3. UPDATE {tab}_pa SET nomor_pam=?, status='on_process'

4.5 Alur Payment Approval Memo (PAM)

Dashboard [SMT | ETF] → Menu: Payment Memo

  → Tab: Draft Memo  → cari PAM No → preview + inline edit → export PDF/Excel
  → Tab: Days of PAM → search-first lazy-load → bulk update 4 SLA dates

9ETF Payment Application — Spesifikasi Detail

9.1 Overview

Modul ini mencatat, melacak, dan memonitor pengajuan pembayaran beasiswa ETF per tipe PA. Berbeda dari modul payment_application (per-memo), modul ini:

  • Mendukung 4 tipe PA: AGRI, APP, LAND, SETF — masing-masing dengan tabel DB sendiri
  • Satu PA mencakup beberapa siswa sekaligus (header + lines)

12Security & Access Control

12.1 Authentication Flow

POST /api/auth/login → verifikasi bcrypt
  → access_token (JWT, expire 1 jam)
  → refresh_token (JWT, expire 7 hari, hash di tabel refresh_tokens)

Setiap request → middleware decode access_token
Token expire   → client gunakan refresh_token untuk token baru
Logout         → revoke refresh_token di DB

12.2 Company Data Isolation

  • Semua tabel bisnis memiliki kolom company_id
  • Setiap query wajib menyertakan WHERE company_id = ? dari session aktif
  • Company aktif disimpan di JWT payload — tidak bisa dimanipulasi client
  • Tidak ada endpoint yang mengembalikan data lintas perusahaan

12.3 API Security

  • CORS dikonfigurasi hanya untuk origin yang diizinkan (localhost / IP LAN)
  • Semua input divalidasi sebelum eksekusi ke SQLite (parameterized query)
  • IDOR guards: setiap resource divalidasi company_id sebelum dikembalikan
  • PAM number collision check real-time via GET /payment-memo/pam/check
