1Executive Summary / Ringkasan Eksekutif

Finance Hub adalah sistem manajemen keuangan internal berbasis web untuk Account Payable, Petty Cash, Beasiswa, dan proses pembayaran yang dijalankan secara lokal (localhost/LAN). Sistem ini dirancang sebagai platform multi-perusahaan untuk dua entitas: Sinar Mas Tjipta dan Eka Tjipta Foundation.

2Problem Statement / Latar Belakang & Tujuan

2.1 Masalah Sebelumnya

  • File Excel terpisah per modul, tersimpan di shared folder lokal (Y:/) tanpa kontrol akses.
  • Tidak ada audit trail — tidak diketahui siapa yang mengubah data dan kapan.
  • File lock conflict ketika Excel dibuka oleh lebih dari satu orang bersamaan.
  • Payment Approval Memo dilakukan manual — tidak ada rekap otomatis lintas modul.
  • Data Sinar Mas Tjipta dan Eka Tjipta Foundation tidak terpisah secara sistematis.

2.2 Solusi

  • Portal terpusat — semua transaksi finansial diinput via web app.
  • Multi-company support — data SMT dan ETF terisolasi penuh di database.
  • Auth berbasis JWT — hanya user yang login yang dapat mengakses sistem.
  • Payment Approval Memo otomatis — rekap dari semua sub-modul dikumpulkan dalam satu tempat.
  • REST API — siap diintegrasikan ke platform induk via JSON API.
  • Export PDF & CSV — laporan dapat diunduh untuk keperluan approval dan arsip.

2.3 Tujuan Sistem

TujuanIndikator Keberhasilan
Digitalisasi data keuangan100% input via web, tidak ada Excel manual
Isolasi data antar perusahaanQuery selalu difilter per company_id
Kontrol aksesTidak ada halaman yang dapat diakses tanpa login
Payment approval terkonsolidasiSemua pembayaran dari semua modul masuk ke PAM
Integrasi-readySemua operasi tersedia via REST API JSON

2.4 Non-Tujuan (Out of Scope v1.0)

  • Integrasi ERP eksternal (SAP, Oracle, dll)
  • Notifikasi email/WhatsApp otomatis
  • Multi-level approval workflow
  • Mobile native app
  • Modul Bank, Account Payable, Advance, Petty Cash, Sponsorship

3.3 User Management

  • Bootstrap: Saat pertama kali dijalankan, sistem otomatis buat akun admin dengan role Releaser dan password default.
  • First login: Wajib ganti password default sebelum dapat mengakses modul manapun.
  • Tambah user: Hanya user dengan role Releaser yang dapat menambah user baru dan assign role-nya.
  • Nonaktif user: User dapat di-nonaktifkan tanpa dihapus — data audit tetap terjaga.

4User Journey Flows / Alur Pengguna

4.1 Login & Company Selection

Buka http://localhost:8080
  → Redirect ke /login (jika belum punya token)
  → Input username + password
  → Verifikasi JWT → redirect ke /select-company
  → Pilih perusahaan: [Sinar Mas Tjipta] | [Eka Tjipta Foundation]
  → Masuk ke Dashboard perusahaan yang dipilih

4.2 Input Beasiswa (ETF)

Dashboard ETF → Menu: Beasiswa
  → Tab: Data Siswa
    → Tambah Siswa Baru → Isi form → Submit
    → Cari Siswa → Edit → Update
  → Tab: Budget
    → Pilih Siswa → Isi Kategori/Semester/Amount → Submit
  → Tab: Payment
    → Pilih Siswa → Isi data pembayaran → Submit
    → Sistem otomatis buat draft entry di Payment Approval Memo
  → Tab: Rekap
    → Filter → Export CSV | Export PDF

4.3 Payment Approval Memo

Dashboard → Menu: Payment Approval Memo
  → Lihat daftar payment draft dari semua modul
  → Pilih payment → tambahkan ke Memo baru
  → Generate Memo Number: PAM/ETF/2026/001
  → Print / Export PDF memo untuk approval
  → Update status: Draft → Submitted → Approved → Paid

8Payment Approval Memo — Spesifikasi Detail

Payment Approval Memo (PAM) adalah hub agregasi pembayaran yang mengumpulkan data payment dari semua sub-modul aktif dan mengkonsolidasikannya menjadi satu dokumen memo pengajuan pembayaran.

8.1 Alur PAM

Payment diinput di sub-modul (Beasiswa / AP / Advance / dll)
  → Entry otomatis masuk sebagai "draft" di PAM
  → Operator review daftar draft payment
  → Operator buat Memo baru → pilih item yang akan diajukan
  → Sistem generate Memo Number: PAM/{COMPANY}/{YYYY}/{SEQ}
      Contoh: PAM/ETF/2026/001, PAM/SMT/2026/015
  → Print / Export PDF memo untuk approval atasan
  → Update status: Draft → Submitted → Approved → Paid

8.2 Status Payment Item

StatusKeterangan
draftPayment baru diinput di sub-modul, belum dimasukkan ke memo
in_memoSudah dimasukkan ke memo, menunggu approval
approvedMemo telah disetujui atasan
paidPembayaran telah dilakukan/dicairkan

8.3 Konten Memo PDF

  • Header: nama perusahaan, nomor memo, tanggal
  • Tabel item: No, Nama Penerima, Bank/Rekening, Keterangan, Amount
  • Sub-total per modul sumber
  • Total keseluruhan
  • Kolom tanda tangan: Dibuat oleh / Disetujui oleh / Tanggal

11Security & Access Control / Keamanan

11.1 Authentication Flow

1. User POST /api/auth/login → verifikasi password dengan bcrypt
2. Jika valid → generate:
   → access_token: JWT, expire 1 jam
   → refresh_token: JWT, expire 7 hari, hash disimpan di refresh_tokens
3. Setiap request API → middleware decode access_token
4. Token expire → gunakan refresh_token untuk dapat access_token baru
5. Logout → revoke refresh_token di database

11.2 Keamanan Data

  • Password di-hash dengan bcrypt (cost factor 12) — tidak ada plain text password di database.
  • JWT secret disimpan di environment variable, bukan di kode.
  • Semua query SQLite menggunakan parameterized query (?) — tidak ada SQL injection.
  • CORS dikonfigurasi hanya untuk origin localhost / IP LAN yang diizinkan.
  • First login: user wajib ganti password default sebelum akses modul.

12Non-Functional Requirements / Batasan & Risiko

12.1 Performance Target

MetrikTarget
Waktu load halaman< 2 detik pada LAN
Response API< 500ms untuk query normal
Concurrent users10–20 user (SQLite write lock per transaksi)
Ukuran databaseEstimasi < 100MB untuk 5 tahun data

12.2 Deployment — Laptop sebagai Server LAN

  [Laptop Server — nyala saat jam kerja]
    └── python app.py  (Flask + waitress, port 8080)
    └── finance_hub.db (SQLite, lokal di laptop)

  [Kolega A] ──browser──→ http://[IP-laptop]:8080
  [Kolega B] ──browser──→ http://[IP-laptop]:8080
  [Kolega C] ──browser──→ http://[IP-laptop]:8080

  Backup harian → copy finance_hub.db ke Y:/ shared folder
  • Laptop server harus tidak sleep/hibernate saat jam kerja (layar boleh mati).
  • Cari IP laptop: ipconfig → IPv4 Address di adapter WiFi/Ethernet.
  • Rekomendasikan IT set static IP agar kolega tidak perlu update alamat setiap hari.
  • Y:/ digunakan sebagai backup destination saja — bukan lokasi database aktif.

12.3 Risiko & Mitigasi

RisikoDampakMitigasi
File finance_hub.db terhapus / corruptKehilangan semua dataBackup harian copy ke Y:/ via Windows Task Scheduler
Laptop server sleep / matiSemua user tidak bisa aksesSet power plan ke "High Performance", matikan sleep saat plugged in
IP laptop berubah (DHCP)Kolega tidak bisa connectSet static IP di network adapter, atau catat IP aktif setiap hari
SQLite write lock jika concurrent write tinggiOperasi simpan lambatAcceptable untuk 10–20 user; upgrade ke PostgreSQL jika perlu scale
JWT secret bocorSemua token bisa dipalsukanSimpan JWT secret di file .env, tidak di-commit ke git
