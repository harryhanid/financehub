1Executive Summary / Ringkasan Eksekutif

Finance Hub adalah sistem manajemen keuangan internal berbasis web untuk Account Payable, Petty Cash, Beasiswa, dan proses pembayaran yang dijalankan secara lokal (localhost/LAN). Sistem ini dirancang sebagai platform multi-perusahaan yang menampung sub-modul finansial dari dua entitas: Sinar Mas Tjipta (SMT) dan Eka Tjipta Foundation (ETF).

Finance Hub menggantikan sistem lama berbasis file Microsoft Excel VBA Macro (DbBeasiswa v2.2.xlsm) yang dioperasikan secara bersama via shared folder Y:/.

2Problem Statement / Latar Belakang & Tujuan

2.1 Masalah Sebelumnya (DbBeasiswa v2.2.xlsm)

  • File lock conflict — hanya 1 orang bisa edit, sisanya Read-only
  • Tidak ada autentikasi — siapapun yang akses Y:/ dapat mengubah data
  • Tidak ada audit trail — tidak diketahui siapa/kapan mengubah data
  • PAM manual — mengisi sheet Detail PAM dan Rangkuman PAM secara manual, lalu mencetak
  • PAM Recon manual — PivotTable harus di-refresh manual setiap perubahan
  • Days of PAM ad-hoc — tanggal SLA diisi di kolom Cat3–Cat7 tanpa validasi
  • ETF PA tidak terstruktur — tidak ada tracking PA dengan 7 SLA dates, tidak ada link ke PAM
  • Tidak ada konfirmasi aksi — simpan/hapus langsung tanpa dialog konfirmasi

2.2 Solusi yang Diimplementasikan

Portal Terpusat

Semua transaksi finansial diinput via web app multi-user

Auth JWT

Login wajib; role-based access per aksi

PAM Otomatis

Draft Memo dari data payment; export PDF/Excel standar

Dynamic Badges

Summary Budget/Payment/Selisih filter-aware real-time

ETF PA Module

4 tipe (AGRI/APP/SML/SETF), 7 SLA dates, auto-link PAM

PAM Integration

Input panel terpadu: baca PA open → save atomik → update PA status

Status Standar

open → on_process → complete di semua modul

Confirm Modal

Global confirmModal() untuk semua aksi kritis

4User Journey Flows / Alur Pengguna

4.1 Login & Company Selection

Buka http://localhost:8080
    → Redirect ke /login (jika belum punya token)
    → Input username + password
    → Verifikasi JWT → redirect ke /select-company
    → Pilih perusahaan: [Sinar Mas Tjipta] | [Eka Tjipta Foundation]
    → Masuk ke Dashboard perusahaan yang dipilih

4.2 Alur Input Beasiswa (ETF)

Dashboard ETF → Menu: Beasiswa
    → Tab: Data Siswa    — CRUD + search/filter + export
    → Tab: Input Budget  — autocomplete siswa, batch Cat1/Cat2/Amount
    → Tab: Data Budget   — filter, dynamic badges (filter-aware), export CSV/PDF
    → Tab: Input Payment — multi-siswa, PAM auto-format, 4 SLA dates
                           → confirmModal → status "open" → auto-create PAM record
    → Tab: Data Payment  — filter, dynamic badges (filter-aware), export CSV/PDF
    → Tab: Report Siswa  — detail per siswa, IPK, rekap, export CSV/PDF

4.3 Alur ETF Payment Application (AGRI/APP/SML/SETF)

Dashboard ETF → Menu: ETF Payment Application
    → Pilih tab: [AGRI] | [APP] | [SML] | [SETF]
    → Daftar PA (satu baris = satu PA header)
        Kolom: PA Number, Jml Siswa, Total, 7 SLA dates, Nomor PAM, Tgl Bayar, Status
    → Buat PA Baru → isi header + baris siswa (autocomplete)
                   → confirmModal → status "open"
    → Edit PA     → update SLA dates, nomor PAM, status
                   → saat status → on_process dan nomor_pam kosong: auto-generate PAM

4.4 Alur PAM Input Integration (ETF PA → Payment Memo)

Payment Approval Memo → Tab: Input PA
    → Pilih Tipe PAM: [AGRI▾] | [APP] | [SML] | [SETF]
    → No. PAM: auto-generate (dapat di-override manual)
        Format validation: PAM-054-AGRI-06-2026
        Collision check: GET /payment-memo/pam/check (saat blur)
    → Tambah Baris siswa (hanya siswa dengan open PA tipe terpilih)
        → Cat1 + Cat2 dipilih:
          1 PA match  → auto-fill Amount, SLA dates
          2+ PA match → PA Picker sub-row muncul (mini-table No PA | Amount)
    → "Simpan PAM {TYPE}" → confirmModal
      Backend (atomik):
        INSERT pam_records (source='etf_{tab}', keterangan)
        INSERT payment_beasiswa (per baris, status='open')
        UPDATE {tab}_pa SET nomor_pam=?, status='on_process'

4.5 Alur PAM End-to-End

Input Payment / Input PA → PAM record auto-created
    → Tab "Daftar PAM": filter source (AGRI/APP/SML/SETF), isi GL Account
    → Tab "Draft Memo": cari PAM No, preview, inline edit, export PDF/Excel
    → Tab "Days of PAM": search-first lazy-load, bulk-update 4 tanggal SLA

Status progression:
    open → (masuk memo) → on_process → (dibayar) → complete

5.3 confirmModal — Global Utility

Semua aksi simpan/update/hapus menggunakan confirmModal() yang didefinisikan di base.html:

confirmModal(message, opts) → Promise<boolean>

opts: {
  type: 'primary' | 'danger',   // biru atau merah
  title: string,
  confirmText: string,
  cancelText: string
}

// Contoh penggunaan
if (!await confirmModal("Simpan data ini?")) return;
if (!await confirmModal("Hapus?", { type: 'danger', confirmText: 'Hapus' })) return;

12Security & Access Control

12.1 Authentication Flow

1. POST /api/auth/login → verifikasi password bcrypt
2. access_token: JWT, expire 1 jam, payload: {user_id, username, company_active}
   refresh_token: JWT, expire 7 hari, di-store hash-nya di tabel refresh_tokens
3. Setiap request → middleware decode access_token
4. Token expire → gunakan refresh_token untuk access_token baru
5. Logout → revoke refresh_token di database

12.2 Company Data Isolation

  • Semua tabel data bisnis memiliki kolom company_id
  • Setiap query wajib menyertakan WHERE company_id = ? dari session aktif
  • Tidak ada endpoint yang mengembalikan data lintas perusahaan
  • Company aktif disimpan di JWT payload — tidak bisa dimanipulasi client

12.3 API Security

  • CORS hanya untuk origin yang diizinkan (localhost / IP LAN)
  • Semua input divalidasi dengan parameterized SQL — tidak ada string concatenation
  • IDOR guards: setiap resource divalidasi kepemilikan company_id
  • Password: bcrypt cost factor 12; wajib ganti default password saat first login
